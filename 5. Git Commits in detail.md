# Keeping your Commits Atomic
When possible, a commit should encompass a single feature, change, or fix. In other words, try to keep each commit focused on a single thing.

This makes it much easier to undo or rollback changes later on. It also makes your code or project easier to review.

**Why does it help with reviews?** If a pull request contains small, distinct commits, reviewers can look at the history step by step rather than wading through a massive wall of mixed changes.

**What if I made a massive commit by mistake?** You can use Git commands like reset to undo the commit and then stage your files separately.

# Past or present tense?

Describe changes in the imperative mood (e.g. "Make X do Y") as if commanding the codebase to change its behaviour.

**Why use the imperative mood for Git commits?** It matches the tone of automated commits generated by Git itself, like "Merge branch".  

**What are some examples of words to avoid?** Avoid words like "fixed", "added", "changing", or "I". Use "Fix", "Add", and "Change".

# A closer look at the Git Log command
To make a long git log more of a summary, you can use:
```bash
git log --oneline
```
# Fixing mistakes with amend

If you make a commit and find a mistake such as a type in the commit message or not adding a file you want to redo the previous commit:

```bash
git commit -m "test message"
git add test.file
git commit --amend
```
# Ignoring files with .gitignore

We can tell git to ignore a file or folder by using .gitignore. It is useful to stop log and setup files from being commited to a repository.

In the root of the repository, create a file called .gitignore.

```bash
	vim .gitignore
```
